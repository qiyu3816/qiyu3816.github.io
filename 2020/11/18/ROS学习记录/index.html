ROS学习记录

​ 初学ROS，全程完全参考Wiki的教程以及网上找的电子书 AGentleIntroductiontoROS （已经上传到群里了），以下只是主要知识点简短的总结以方便自己复习。

一、安装ROS

​ 按照Wiki ROS教程安装melodic发行版的ROS。

二、入门概述

1）功能包/软件包（Packages）

​ 在ROS中，所有软件都被组织为软件包的形式，称为ROS软件包或功能包，有时也简称为包。ROS 软件包是一组用于实现特定功能的相关文件的集合，包括可执行文件和其他支持文件。比如，可执行文件turtlesim_node和turtle_teleop_key都属于 turtlesim包。

​ 查看软件包和定位软件包： rospack list

​ 软件包目录（package.xml） ，每个程序包都有这么一个清单文件定义一些关于包的细节。其实，软件包的定义就是——任何ROS能找到且包含package.xml文件的目录就是软件包目录。

​ 找到一个软件包的目录 ： rospack find <package-name>

​ 查看软件包 ： rosls <package-name>

​ 访问软件包目录 ： roscd <package-name>

2）节点管理器

​ ROS 的一个基本目标是使机器人专家设计的很多称为节点 （node）的几乎相对独立的小程序能够同时运行。为此，这些节点必须能够彼此通信。ROS 中实现通信的关键部分就是ROS节点管理器。

​ 启动节点管理器 ： roscore

​ 节点管理器应该在使用ROS的全部时间内持续运行。一个合理的工作流程是在一个终端启动roscore，然后打开其他终端运行其他程序。除非你已经完成ROS的相关工作，否则一般没有理由终止 roscore 命令。当结束时，可以通过在roscore终端键入Ctrl-C停止节点管理器。

3）节点（Nodes）

​ ROS程序的运行实例被称为节点（node）。

​ 在turtlesim的例子中，第一个节点是可执行文件turtlesim_node的实例化。这个节点负责创建turtlesim窗口和模拟海龟的运动。第二节点是可执行文件turtle_teleop_key的实例化。teleop 是 teleoperation（遥操作）的缩写，是指人通 过在远程发送运动指令控制机器人。这个节点的作用是捕捉方向键被按下的事件，并将方向键的按键信息转换为运动指令，然后将命令发送到turtlesim_node节点。

​ 启动节点 ： rosrun <package-name> <executable-name>

​ 查看节点列表 ： rosnode list

​ 关于此列表：

  • rosout节点是一个特殊的节点，通过roscore自动启动。其作用有点类似于控制台程序中使用的标准输出（即 std::cout）。
  • 节点名并不一定与对应可执行文件名称相同。

​ 查看节点 ： rosnode info <node-name> ，显示是发布者或订阅者等详细信息。

​ 终止节点 ： rosnode kill <node-name> ，全清 rosnode cleanup

4）话题和消息

​ ROS节点之间进行通信所利用的最重要的机制就是消息传递。 在ROS中，消息有组织地存放在话题里。消息传递的理念是： 当一个节点想要分享信息时，它就会发布(publish)消息到对应的一个或者多个话题；当一个节点想要接收信息时，它就会订阅 (subscribe)它所需要的一个或者多个话题。ROS节点管理器负责确保发布节点和订阅节点能找到对方；而且消息是直接地从发布节点传递到订阅节点，中间并不经过节点管理器转交。

​ 打印消息内容 ： rostopic echo <topic-name> ，比如可以打印虚拟海龟的信息就用 rostopic echo /turtle1/cmd_vel 。

​ 测量发布频率 ： rostopic hz <topic-name> rostopic bw <topic-name> 用来测量消息发布的频率以及这 些消息所占用的带宽。

​ 查看话题 ： rostopic info <topic-name> 显示话题的消息类型（Type），发布者和订阅者。

  • 首先声明头文件ros/ros.h。

  • 然后ros::init函数初始化ROS客户端库。请在程序的起始处调用 一次该函数。函数最后的参数是一个包含节点默认名的字符串。

  • ros::NodeHandle（节点句柄）对象是你的程序用于和ROS系统交互的主要机制 。创建此对象会==将你的程序注册为ROS节点管理器的节点==。最简单的方法就是在整个程序中只创建一个NodeHandle对象。

    在内部，NodeHandle 类维护一个引用计数，仅仅在第一个NodeHandle对象创建时才会在节点管理器注册新的节点。同样，只有当所有的NodeHandle对象都销毁后，节点才会 注销。

  • ROS_INFO_STREAM宏将生成一条消息，且这一消息被发送到不同的位置，包括==控制台窗口==。

4）发布者程序（pubvel.cpp）

​ 这里只介绍相对hello_world.cpp添加的流程。

  • 这里声明头文件还要添加程序中每个用到的消息类型功能包 #include<package_name/type_name.h> ，功能包名应该是定义消息类型的包的名称， 而不一定是你自己的包的名称。比如要发布类型为 geometry_msgs/Twist 的消息 #include <geometry_msgs/Twist.h> 。

  • 创建发布者对象， ros::Publisher pub = node_handle.advertise( topic_name, queue_size); 其中：

    node_handle是在程序最初创建的；

    topic_name是想要发布到的话题；

    queue_size是一个整数，表示这个发布者发布的消息序列的大小。通常设置成1000比较合适。是因为大多数情况下，消息必须发送到其他节点。这个通信过程可能是耗时的，特别是相比于创建消息所需要的时间。

    注意：==最好为每个话题创建一个发布者，并在执行过程中一直是用它==。

  • 创建并填充消息对象，pubvel.cpp中声明geometry_msgs::Twist创建消息对象实例，然后用(“.”)符号访问各个域进行填充。

  • 发布消息，pub.pubish(msg) ，用发布者对象调用此函数并以消息为输入参数。

  • 节点是否停止工作的检查， 若节点被 rosnode kill 了，或终端Ctrl-C了，或程序中调用了 ros::shutdown() ，或启动了一个相同程序的新实例，ros::ok() 会返回false。

  • 控制消息发布频率，ros::Rate rate(2);

    ​ 这个对象控制循环运行速度，其构造函数中的参数以赫兹（Hz） 为单位，即每秒钟的循环数。这个例子创建了旨在规范每秒钟执 行两个迭代循环的速率对象。

    ​ 邻近每次循环迭代的结尾，调用此对象的 sleep 方法： rate.sleep();

    ​ 每次调用此方法时就会在程序中产生延迟。延迟的持续时间被用来阻止循环的迭代速率超过指定的速率。==没有这种控制，程序会 以计算机允许的最快速度发布消息==，这样会占满发布和订阅的序列。

  • 定义输出格式，用宏ROS_INFO_STREAM，可以看到代码中的输出和C++常用的(“<<”)符没有太大差别。

  • 编译这个pubvel.cpp时CMakeLists.txt需要作出添加 find_package(catkin REQUIRED COMPONENTS roscpp geometry_msgs) 。

    package.xml需要添加 <build_depend>geometry_msgs</build_depend> <run_depend>geometry_msgs</run_depend> 。

  • 执行和之前无异。

    5）订阅者程序（subpose.cpp）

​ 这里只介绍相对hello_world.cpp添加的流程。

  • 编写回调函数，这个回调函数用于控制接收到话题的消息后所作的动作。

    void function_name(const package_name::type_name &msg) { . . . }

    输入参数就是一个消息。

  • 创建订阅者对象， ros::Subscriber sub = node_handle.subscribe (topic_name,queue_size, pointer_to_callback_function);

    其中：

    node_handle仍是程序开始创建的ros::NodeHandle对象；

    此后三个参数除了多了个回调函数指针外与发布者无异。

    注意：queue_size是本订阅者的消息队列大小，也是整数。当新的消息到达时，它们会被保存在这个队列中，直到 ROS 有机会去执行相应的回调函数。此参数表示 ROS 在队列中 同一时刻可以存储的消息的最大值。==如果新消息到达时队列已满，最早到达的还没有被处理的消息将会被丢弃以便腾出空间来。== 通过如下两个方法减少订阅者队列溢出的可能性：

    （1）通过调用 ros::spin 或者 ros:spinOnce 确 保允许回调发生；

    （2）减少每个回调函数的计算时间。

  • 给ROS控制权，==只有当我们明确给ROS许可时， 它才会执行我们的回调函数== 。

    ros::spinOnce();

    这个代码要求 ROS 去执行所有挂起的回调函数，然后将控制权限返回给我们。另一个方法：

    ros::spin();

    这个方法要求 ROS 等待并且执行回调函数，直到这个节点关机。 换句话说，ros::spin()大体等于这样一个循环：

    while( ros::ok() ) { ros::spinOnce(); }

    使用 ros::spinOnce()还是使用 ros::spin()的建议如下：你的程序除了响应回调函数，还有其他重复性工作要做吗？如果答案是 “否”，那么使用 ros::spin()；否则，合理的选择是写一个循环，做其他需要做的事情，并且周期性地调用 ros::spinOnce()来处理回调。

  • subpose.cpp的编译只需要在CMakeLists.txt添加可执行文件声明即可

    四、日志消息

1）严重级别

​ ROS日志系统的核心思想， 就是使程序生成一些简短的文本字符流，这些字符流便是日志消息。ROS中，日志消息分为五个不同的严重级别，也可简称为严重性或者级别。按照严重性程度递增，这些级别有：

DEBUG

INFO

WARN

ERROR

FATAL

2）生成日志消息

​ 从C++代码生成日志消息。 生成简单的日志消息总共有五个基本的 C++宏用来产生日志消 息，其中每个宏对应一个严重级别：

ROS_DEBUG_STREAM(message);

ROS_INFO_STREAM(message);

ROS_WARN_STREAM(message);

ROS_ERROR_STREAM(message);

ROS_FATAL_STREAM(message);

​ 其中各个宏的参数 message 可以处理 C++中标准输出流（ostream） 中的各种表达式，比如 std::cout。这包括在 int 或者 double 这种基本数据类型上使用插入操作符（<<），以及已经重载这个操作符的复合数据类型，或者是标准流操作符，比如 std::fixed， std::setprecision 或者 std::boolalpha。

3）生成一次性日志消息

ROS_DEBUG_STREAM_ONCE(message);

ROS_INFO_STREAM_ONCE (message);

ROS_WARN_STREAM_ONCE (message);

ROS_ERROR_STREAM_ONCE (message);

ROS_FATAL_STREAM_ONCE (message);

​ 程序运行中第一次到达这些宏时，它们将产生与相应的无 -ONCE 后缀版本宏一样的日志消息。第一次执行后，这些声明将不再发挥作用。

4）生成频率受控的日志消息

ROS_DEBUG_STREAM_THROTTLE(interval, message);

ROS_INFO_STREAM_THROTTLE(interval, message);

ROS_WARN_STREAM_THROTTLE(interval, message);

ROS_ERROR_STREAM_THROTTLE(interval, messge);

ROS_FATAL_STREAM_THROTTLE(interval, message);

​ 参数 interval 是 double 类型的，它表示以秒为单位的时间量，这 是相邻日志消息出现的最小时间间隔。

5）查看日志消息

​ 日志消息有三个目的地：控制台，/rosout话题，日志文件。

​ 查看/rosout消息的方式：rostopic echo /rosout 或 rqt_console

6）日志文件

​ 生成日志文件作为/rosout话题回调函数的一部分，该节点可以将日志消息作为一行写入到一个日志文件，文件名类似于： ~/.ros/log/run_id/rosout.log

​ 运行标识码（run_id）是一个通用唯一识别码（UUID），它在节点管理器开始运行时基于你计算机的 MAC 地址和当前的时间生成。

​ 查看运行标识码，通过检查 roscore 生成的输出，在靠近输出末端的位置，可以看到与下面内容类似的一行 setting /run_id to run_id ；或通过以下命令向节点管理器询问当前的 run_id： rosparam get /run_id 。

​ 检查和清楚日志文件，分别用 rosclean check rosclean purge ，ROS不会自动清理日志文件，但在其超过1G时会发出提醒。可以自己清理它们。

五、启动文件

​ ROS 提供了一 个同时启动节点管理器（master）和多个节点的途径，即使用启动文件（launch file）。其基本思想是在一个XML格式的文件内将需要同时启动的一组节点罗列出来。

1）：

  • 插入根元素\…\，每个启动文件的其他元素都应该包含在这两个标签之内。

  • 启动节点

    \<node

    pkg=”package-name”

    type=”executable-name”

    name=”node-name”
    />

    其中：

    pkg 和 type属性定义了 ROS 应该运行哪个程序来启动这个节点。这些和 rosrun 的两个命令行参数的作用是一致的，即给出功能包名和可执行文件的名称；

    name 属性给节点指派了名称，它将覆盖任何通过调用 ros::init 来赋予节点的名称。

  • 在控制台中输出信息，对于某个单独的节点，只需在节点元素中配置 output 属性就可以达到该目的： output="screen"

  • 请求复位，当节点停止的时候， roslaunch 会重新启动该节点： respawn="true"

  • 必要节点（requiring node） ，复位之外的另一种策略是将一个节点声明为必要节点： required=”true”

    当一个必要节点终止的时候，roslaunch 会终止所有其他活跃节点并退出。

  • 为节点维护独立的窗口，对于依赖控制台输入的节点来说， 例如 turtle_teleop_key，更倾向于为节点保留独立的终端，可以使用对节点元素启动前缀（launch-prefix）属性：

    Launch-prefix="command-prefix" 。

    例如 launch-prefix="xetrm-e"

  • 在命名空间内启动节点，节点元素支持对其配置ns命名空间属性。

  • 包含其他文件，如果想在启动文件中包含其他启动文件的内容（包括所有的节点和参数），可以使用包含（include）元素:

    <include file="path-to-launch-file">

    此处 file 属性的值应该是我们想包含的文件的完整路径。由于直接输入路径信息很繁琐且容易出错，大多数包含元素都使用 查找（find）命令搜索功能包的位置来替代直接输入路径：

    <include file=""$(find package-name)/launch-file-name">

    也同样支持ns命名空间参数。

  • 执行启动文件，roslaunch package-name launch-file-name 。添加-v参数可以输出详细信息。

    2）启动参数（launch arguments）

​ 为了使启动文件便于配置，roslaunch还支持启动参数，有时也简称为参数甚至args，其功能有点像可执行程序中的局部变 量。

​ 为了区分，parameter 是运行中的 ROS 系统使用的数值，存储在参数服务器（parameter server）中，每个活跃的节点都可 以通过 ros::param::get 函数来获取 parameter 的值，用户也 可以通过 rosparam 来获得 parameter 的值。 而 argument 只在启动文件内才有意义；他们的值是不能被节点直接获取的。

  • 声明参数，<arg name="agr-name"> 。

  • 参数赋值 ，roslaunch package-name launch-file-name arg-name:=arg-value

    或 <arg name=”arg-name” default=”arg-value”/>

    或 <arg name=”arg-name” value=”arg-value”/>

    两者的唯一区别在于命令行参数可以覆盖默认值 default，但是不能覆盖参数值 value。由 value 设置的参数值是不能更改的。

  • 获取参数值，一旦参数值被声明并且被赋值，你就可以利用下面的 arg 替换（arg substitution）语法来使用该参数值了： $(arg arg-name)

  • 向包括的启动文件中发送参数值，只需要在\和\之间加入arg元素即可。但==在 include 标签之间的参数是属于被包含文件的，而不是它们出现的启动文件==。

​ 组(group)元素，提供了一种在大型启动文件内管理节点的便捷方式。

​ 组可以把若干个节点放入同一个命名空间内。

​ \

​ \</group>

​ 如果一个组元素内的某个节点有它自己的命名空间属性， 并且其名称是（原则上也应该是）相对名称，那么该节点将会在一个默认命名空间内启动，这个默认的命名空间是将此节点命名空间嵌入到组元素命名空间的结果。

六、参数

​ ROS还提供另一种参数 （parameters）机制用于获取节点的信息。其主要思想是使用集中 参数服务器（parameter server）维护一个变量集的值，包括整数、 浮点数、字符串以及其他数据类型，每一个变量用一个较短的字符串标识 。由于允许节点主动查询其感兴趣的参数的值，它们适用于配置那些不会随时间频繁变更的信息。

1）通过命令行获取参数

​ rosparam list ，输出：

/rosdistro

/roslaunch/uris/host_donatello__38217

/rosversion

/run_id

参数服务器是节点管理器的一部分， 因此，它总是通过 roscore 或者 roslaunch 自动启动。

​ 查询参数，rosparam get <parameter_name>

​ 设置参数，rosparam set <parameter_name> <parameter_value> ，也可以以YAML字典形式设置多个参数。

​ 创建和加载参数文件，rosparam dump filename namespace rosparam load filename namespace 。

2）使用C++获取参数

​ void ros::param::set(parameter_name, input_value);

​ bool ros::param::get(parameter_name, output_value);

​ 两个函数分别用于设置和获取参数。

3）启动文件中设置参数

​ <param name="param-name" value="param-value" />

​ 上面的元素设置参数，以下则读取参数：

​ <rosparam command="load" file="path-to-param-file" />

七、服务

​ 服务调用（service calls）是另一种用于通信的方法，与发布订阅的区别主要在于两点：

​ 服务调用是==双向的==，一个节点给另一个节点发送信息并等待响应，因此信息流是双向的；

​ 服务调用实现的是==一对一通信==。

​ 其过程是一个客户端（client）节点发送一些称为请求（request） 的数据到一个服务器（server）节点，并且等待回应。服务器节点接收到请求后，采取一些行动（计算、配置软件或硬件、改变 自身行为等），然后发送一些称为响应（response）的数据给客户端节点。 请求和响应数据携带的特定内容由服务数据类型（service data type）来决定，它与决定消息内容的消息类型是类似的

1）从命令行查看和调用服务

​ 列出所有服务，rosservice list ，输出：

/clear

/kill

/reset

/rosout/get_loggers

/rosout/set_logger_level

/spawn

/turtle1/set_pen

/turtle1/teleport_absolute

/turtle1/ teleport_relative

/turtlesim/get_loggers /turtlesim/set_loggers_level

​ 很多 ROS 服务总的来讲可以划分为两个基本类型。

​ 1）一些服务，例如上表中 get_loggers 和 set_logger_level 服务， 是用来从特定的节点获取或者向其传递信息的。

​ 2）其他服务表示更一般的不针对某些特定节点的服务。

​ 查找提供服务的节点，rosservice <node> <service-name> 。

​ 查找服务的数据类型， rosservice info <service-name> ，例如输出：

​ Node: /turtlesim

​ URI: rosrpc://Donatello:47441

​ Type: turtlesim/Spawn

​ Args: x y theta name

​ 查看服务数据类型，rossrv show <service-data-type-name> ，例如：

​ 从命令行调用服务，rosservice call <service-name> <request-content> 。

  • 声明请求和响应的类型，#include<package_name/type_name.h> ，例如 #include<turtlesim/Spawn.h>

  • 调用服务，bool success = service_client.call(request,reponse); ，这个方法实际上完成了定位服务器节点、传输请求数据、等待响应和存储响应数据等一系列工作。返回一个布尔值来表明服务调用是否成功完成。

3）服务器程序（pubvel_toggle.cpp）

  • 编写服务回调函数，bool function_name( package_name::service_type::Request &req), package_name::service_type::Response &resp) ) { … } 输入参数为一个请求和一个响应对象。
  • 创建服务器对象，ros::ServiceServer server = node_handle.advertiseService( service_name,pointer_to_callback_function); ，与订阅者类似。
  • 提供 ROS 控制权，只有在我们通过 ros::spin()或 者 ros::spinOnce()请求时，ROS 才会执行回调函数。
